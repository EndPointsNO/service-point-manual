# Arbeidsflyt for innhold

Bruk denne siden som fast prosess når manualen skal oppdateres. Målet er at innhold, screenshots, hurtigkort og deploy alltid holder samme kvalitet.

## Når brukes dette?

Når dere skal:

- oppdatere en eksisterende prosedyre
- legge til en ny side
- erstatte TODO-screenshots med faktiske skjermbilder
- rette feil som brukere eller butikk melder inn

## Mål

Alle endringer skal være små, kontrollerbare og publiserbare uten at sidebaren, lenker eller screenshots faller ut av sync.

## Standard arbeidsflyt

1. Finn riktig side i `docs/` og avklar hva som faktisk har endret seg i PDA eller rutinen.
2. Oppdater teksten på den aktuelle docs-siden med vanlige overskrifter, punktlister og bilder.
3. Hvis prosedyren endrer steg eller skjermbilder:
   oppdater også filnavn, bildehenvisninger og [Screenshot backlog](./screenshot-backlog).
4. Hvis siden er mye brukt i drift:
   oppdater også [Hurtigkort](./hurtigkort) og eventuelt hurtigvalg på startsiden.
5. Hvis du legger til en ny side:
   legg den inn i `sidebars.ts` og legg til relaterte lenker fra nærliggende sider.
6. Kjør `npm run build`.
7. Commit og push til `main` for automatisk deploy til GitHub Pages.

## Tre typiske endringer

### 1. Liten tekstendring

Bruk når:
- teksten er uklar
- et kontrollpunkt må presiseres
- det er en liten feil i navn eller menytekst

Gjør dette:
1. Oppdater siden
2. Kjør `npm run build`
3. Push

### 2. Prosedyreendring

Bruk når:
- PDA-flyten har fått nye steg
- et kontrollpunkt er endret
- samme side trenger nye screenshots

Gjør dette:
1. Oppdater prosedyresiden
2. Oppdater relaterte sider
3. Oppdater screenshot backlog
4. Oppdater hurtigkort hvis prosessen er blant de viktigste
5. Kjør `npm run build`
6. Push

### 3. Ny side

Bruk når:
- dere får en ny operativ rutine
- en eksisterende side bør splittes i to
- det mangler en egen oversikt eller feilside

Gjør dette:
1. Opprett ny `.mdx`-fil i `docs/`
2. Følg standard mal:
   `Når brukes dette?`, `Mål`, `Steg for steg`, `Kontrollpunkt`, `Vanlige feil`, `Relaterte sider`
3. Legg siden inn i `sidebars.ts`
4. Legg til lenker fra nærliggende sider
5. Kjør `npm run build`
6. Push

## Hva skal oppdateres hvor?

| Hva du vil endre | Hvor du endrer det |
| --- | --- |
| Hurtigvalg på forsiden | `docs/intro.mdx` |
| Sidebar | `sidebars.ts` |
| Prosedyresteg | aktuell side i `docs/` |
| Teknisk layout for forside | `src/components/ManualLayout/` |
| Screenshot-fil | `static/img/pda/` |
| Manglende screenshots | `docs/screenshot-backlog.mdx` |
| Printvennlig oppsummering | `docs/hurtigkort.mdx` |

## Kvalitetssjekk før publisering

- Er sidebaren fortsatt riktig?
- Er alle interne lenker gyldige?
- Har alle steg riktig bilde eller TODO-plassholder?
- Stemmer navnekonvensjonen på screenshots?
- Må hurtigkort oppdateres?
- Er siden skrevet i enkel markdown uten tekniske komponenter?

## Minimumskrav før push

```bash
npm run build
```

Bygget skal være grønt før endringer pushes.
